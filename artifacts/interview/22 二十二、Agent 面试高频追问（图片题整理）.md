# 二十二、Agent 面试高频追问（图片题整理）

题目数：44

本章针对三张图片中的一面、二面、三面问题生成面试回答。回答默认面向 Agent 开发、LLM 工程、AI Coding 和企业级落地方向，强调工程判断、选型逻辑和可落地表达。

## 一面

### 01 MCP、CLI、Skills 三者核心区别是什么？

**考点**

三者都能给 Agent 扩展能力，但抽象层不同：MCP 是协议，CLI 是操作入口，Skills 是可复用能力包。

**回答要点**

- **MCP**：Model Context Protocol，本质是工具和上下文接入协议。它让外部系统以标准方式暴露 tools、resources、prompts，适合跨工具、跨平台集成。
- **CLI**：命令行工具，本质是执行入口。Agent 通过 shell 调用已有工程工具，例如 `git`、`npm`、`pytest`、`kubectl`，优点是复用成熟生态。
- **Skills**：面向 Agent 的能力封装，本质是“如何使用某类工具/知识/流程”的说明书和脚手架。它通常包含规则、脚本、模板和工作流。

**面试表达**

> MCP 解决“工具如何标准化接入”，CLI 解决“如何调用已有系统能力”，Skills 解决“Agent 如何稳定掌握一类任务方法”。工程里三者经常组合使用：MCP 做跨系统协议，CLI 复用成熟工具链，Skills 把经验沉淀成可复用流程。

---

### 02 为什么业界更倾向用 CLI 工具而非 MCP 工具？

**考点**

不是 MCP 不重要，而是 CLI 更成熟、更稳定、更低成本。

**回答要点**

- CLI 生态已有几十年积累，覆盖面广、语义稳定、可脚本化、易调试。
- CLI 输出虽然不总是结构化，但真实工程工具大多先有 CLI，再有 API 或 MCP。
- MCP 的优势是标准化和上下文治理，但生态仍在扩展，很多企业内部工具还没有 MCP server。
- 对 coding agent 来说，`rg`、`git`、`pytest`、`npm`、`docker` 这些 CLI 能直接完成核心闭环。

**面试表达**

> 当前更常用 CLI，不是因为 MCP 价值低，而是因为 CLI 是存量工程世界的事实标准。MCP 更像未来的能力协议层，适合做工具治理和跨平台集成；CLI 更像今天最可靠的执行层，适合快速落地。

---

### 03 MCP 和 CLI 在上下文加载、上下文占用体积上有什么差异？

**回答要点**

- MCP 通常会把工具 schema、描述、资源索引暴露给模型，优点是可发现、可治理，缺点是工具多时上下文占用明显。
- CLI 通常只把命令执行结果放入上下文，工具说明可由系统提示或 skill 控制，上下文更按需。
- MCP 更适合结构化工具路由；CLI 更适合确定性执行和按需读取。
- 大规模工具场景下，不能把所有 MCP 工具一次性塞给模型，需要工具路由、懒加载、分组和检索。

**面试表达**

> MCP 的上下文成本主要来自工具描述和 schema；CLI 的上下文成本主要来自命令输出。MCP 更适合“模型自己选工具”，CLI 更适合“系统明确调用某个工具”。工具数量上来后，两者都需要上下文预算管理。

---

### 04 工具数量变多后，容易出现路由不准、参数解析错误怎么解决？

**回答要点**

- 工具分层：先选 tool category，再选具体 tool。
- 工具描述标准化：名称、适用场景、反例、参数 schema、返回协议统一。
- Top-K 工具检索：按任务意图检索少量候选工具，不全量暴露。
- Few-shot 工具调用示例：给高频工具配典型调用样例。
- 执行前校验：参数用 Pydantic/JSON Schema 校验，失败后让模型修正。
- 高风险工具加 policy 和 approval。

**面试表达**

> 工具多了以后不能只靠模型在一百个函数里自由选择，而要引入 Tool Registry、Tool Router 和 Policy 层。我的做法是先把工具按领域分组，再通过意图分类或检索召回候选工具，最后用 schema 校验兜底。

---

### 05 高频工具调用使用 Few-Shot 示例能带来什么效果？

**回答要点**

- 降低参数格式错误，尤其是复杂 JSON 参数。
- 明确工具适用边界，减少误调用。
- 提升多步工具链稳定性，例如“先查状态，再写入，再校验”。
- 让模型学习错误恢复方式，例如参数校验失败后如何重试。

**面试表达**

> Few-shot 的价值不是让模型记住某个答案，而是让模型学习“这个工具应该怎么被调用”。对高频工具，示例比长描述更有效，尤其能降低参数结构错误和工具选择错误。

---

### 06 MCP 工具从几十扩充到上百个，会出现哪些问题？通用解决方案是什么？

**回答要点**

典型问题包括上下文膨胀、工具选择混乱、权限难治理、响应变慢、schema 冲突、工具版本不可控。

通用方案：

- 工具目录分层：domain/category/tool。
- 动态加载：只把当前任务相关工具注入上下文。
- 工具检索：通过 embedding/关键词/规则召回候选工具。
- 工具元数据治理：risk、owner、version、timeout、approval。
- 工具审计：记录 tool_start/tool_end、参数脱敏、结果摘要。

**面试表达**

> MCP 工具规模上来后，本质问题从“工具接入”变成“工具治理”。核心不是把更多工具塞给模型，而是建立工具路由、权限、版本、审计和上下文预算机制。

---

### 07 企业大规模落地 MCP，如何解决工具爆炸、上下文过长的问题？

**回答要点**

- 建立企业级 Tool Catalog，而不是每个团队各自暴露工具。
- 按业务域、权限域、风险等级做工具分组。
- 对模型只暴露少量候选工具，候选由 router 召回。
- 工具文档和示例外置，通过 RAG 按需加载。
- 对工具调用加 trace_id、run_id、user/session 关联，便于审计。

**面试表达**

> 企业级 MCP 不能简单等同于“大家都写 server”。它需要一个中心化工具目录和分布式执行能力：目录负责发现、权限、版本和描述，执行层负责调用、审计、超时和脱敏。

---

### 08 基于 RAG + MCP 做工具路由网关，有哪些代表性框架？

**回答要点**

可以参考三类实现：

- Agent 编排框架：LangGraph、AutoGen、Semantic Kernel，用于控制状态、节点和工具调用。
- MCP 生态：官方 MCP SDK、Cursor/Claude Desktop 等 MCP client/server 形态。
- 检索与路由层：LlamaIndex、LangChain Retriever、向量数据库，用于工具说明检索和候选召回。

**面试表达**

> RAG + MCP 的工具路由网关可以理解为“工具说明也进入检索系统”。用户任务先检索相关工具，再把少量工具 schema 注入模型，由模型做最终选择。LangGraph 适合做状态编排，MCP 适合做工具协议，RAG 适合做候选召回。

---

### 09 OpenClaw、Hermes 这类开源框架，针对 MCP 痛点做了哪些优化？

**回答要点**

这类框架的共同方向通常是工具路由、上下文压缩、工具市场、权限审批、执行审计和多工具调度。具体项目能力要以版本为准，不宜在面试中夸大。

可学习的设计点：

- 工具能力用元数据描述，而不是只有 name/description。
- 工具调用链路可观测，包括参数、结果、耗时、错误。
- 对工具集合做动态筛选，减少上下文注入。
- 提供开发者体验，例如本地调试、mock、回放。

**面试表达**

> 我会把这类框架当作设计参考，而不是直接照搬。它们有价值的地方在于把 MCP 从“协议接入”推进到“工具治理平台”：路由、权限、审计、上下文控制和开发者体验。

---

### 10 AI 编程中 Rule 规则和普通 Prompt 的核心区别是什么？

**回答要点**

- Prompt 是单次任务指令，偏即时。
- Rule 是持久化、可复用、可作用域化的工程约束。
- Rule 更适合编码规范、架构边界、测试要求、禁止事项。
- Prompt 更适合当前任务目标、临时偏好、一次性说明。

**面试表达**

> Rule 是工程制度，Prompt 是临时需求。AI Coding 里真正稳定的不是每次把提示写长，而是把项目规范沉淀成规则，让 Agent 每次进入代码库时都自动遵守。

---

### 11 大量 Rule 规则时，如何实现分类、检索、按需加载？

**回答要点**

- 按 scope 分类：全局、项目、目录、文件类型、任务类型。
- 加 metadata：description、globs、priority、always_apply。
- 建立 rule index，通过任务意图和文件路径检索相关规则。
- 控制冲突：高优先级规则覆盖低优先级规则，冲突时显式报错。
- 定期治理：删除重复、过期、不可执行的规则。

**面试表达**

> Rule 多了以后也会变成上下文污染，所以要像管理代码一样管理规则：有作用域、有优先级、有检索、有冲突处理。好的规则系统不是全量加载，而是按任务和文件路径动态加载。

---

### 12 Agent 通用评测机制该如何设计？包含哪些维度？

**回答要点**

评测要覆盖结果、过程和成本：

- 任务成功率：是否达成目标。
- 工具调用准确率：选对工具、参数正确、顺序正确。
- 事实准确率：是否幻觉、是否引用可靠来源。
- 安全性：是否越权、泄露隐私、执行高风险动作。
- 稳定性：多次运行一致性、失败恢复能力。
- 成本和延迟：token、工具耗时、总时延。
- 可观测性：是否能回放 run timeline。

**面试表达**

> Agent 不能只评最终答案，因为过程可能已经造成副作用。我的评测会分三层：最终结果是否正确，执行过程是否合规，成本和稳定性是否可接受。

---

### 13 AutoGPT、Codex 这类经典框架的优缺点和落地局限是什么？

**回答要点**

- AutoGPT 类框架优点是自主规划、工具循环、原型快；缺点是不可控、容易发散、成本高。
- Codex/AI coding agent 类工具优点是深度理解代码、能执行测试和修改文件；缺点是依赖 sandbox、权限、上下文质量和审查流程。
- 落地局限主要在：状态管理、审批、回放、权限、长期记忆、工具审计、可重复测试。

**面试表达**

> 早期 AutoGPT 证明了 Agent loop 的可能性，但生产落地不能只靠“自主循环”。真正要补的是 runtime：状态机、事件日志、审批、取消、回放、权限和评测。

---

### 14 Claude Code 对比 Cursor，核心优势和代际差异在哪？

**回答要点**

- Claude Code 更偏终端里的 agentic coding 工具，适合长任务、命令执行、代码库理解和连续修改。
- Cursor 更偏 IDE 集成，强调编辑器内体验、规则系统、上下文选择、MCP 和交互式改代码。
- Claude Code 的优势常在模型推理和长链执行；Cursor 的优势常在 IDE 体验、代码导航、局部编辑和团队规则沉淀。
- 选型不能只看模型，要看工作流：终端自动化选 Claude Code 类，编辑器协作选 Cursor 类。

**面试表达**

> 我会把 Claude Code 理解成“终端里的 coding agent”，把 Cursor 理解成“IDE 里的 AI 工程环境”。二者不是完全替代关系，差异在交互入口、上下文组织和工程工作流。

### 02 单 Agent 和多 Agent 的选型核心原则是什么？

**回答要点**

- 默认从单 Agent 开始，因为链路短、调试简单、状态一致。
- 当任务天然可并行、工具集明显不同、权限边界不同、角色能力差异大时，再拆多 Agent。
- 多 Agent 的代价是通信、状态同步、结果汇总、冲突处理和 trace 复杂度。

**面试表达**

> 我的原则是“先单后多”。只有当拆分能明确降低复杂度或提升吞吐时，才引入多 Agent。否则多 Agent 很容易把一个模型问题变成分布式系统问题。

---

### 03 多 Agent 适合哪些场景，不适合哪些场景，选型前置逻辑是什么？

**适合**

- 子任务独立并可并行，例如调研、代码审查、测试生成。
- 角色差异明显，例如 planner、coder、reviewer、tester。
- 权限需要隔离，例如只读分析 Agent 和可写执行 Agent。

**不适合**

- 简单问答、短任务、强共享上下文任务。
- 结果强一致、状态频繁交互的任务。
- 工具和上下文高度重叠的任务。

**面试表达**

> 多 Agent 的前置判断是：拆开以后通信成本是否低于单 Agent 的上下文成本。如果不是，就不应该拆。

### 04 大模型领域 SFT 训练数据集一般怎么设计构建？

**回答要点**

- 明确目标能力：问答、代码、工具调用、领域知识、格式遵循。
- 设计数据 schema：instruction、input、output、metadata、source、quality_score。
- 数据来源：真实业务样本、专家标注、规则生成、模型合成、人类修订。
- 数据过滤：去重、脱敏、事实校验、格式校验、低质样本过滤。
- 划分训练/验证/测试集，避免数据泄露。

**面试表达**

> SFT 数据不是越多越好，而是要围绕目标能力设计。一个好数据集要有明确任务分布、质量标签、覆盖边界样本，并且能被评测集验证。

### 05 复盘一个 Agent 类项目，重新落地会做哪些优化？

**回答要点**

- 从一开始建立 thread/run/event 模型，保证可回放。
- 工具调用统一经过 Tool Registry，记录 metadata 和审计。
- 加 Run 状态机：queued/running/waiting_approval/cancelled/completed/failed。
- 记忆系统分层：working memory、semantic RAG、episodic summary。
- 前端优先做 chat + timeline，便于用户理解 Agent 在做什么。
- 增加评测脚本，覆盖 normal/cancel/approval/tool failure。

**面试表达**

> 我不会先追求更多工具，而会先把 runtime 闭环做稳：状态、事件、回放、取消、审批、审计和评测。没有这些，Agent demo 看起来能跑，但出了问题无法定位。

### 06 DeepSeek V4、Claude 4 系列使用体验有什么差异？

**回答要点**

需要客观区分“模型能力”和“工程接口”：

- DeepSeek V4 Pro 走 OpenAI-compatible API，thinking mode 通过 `thinking` 和 `reasoning_effort` 控制；多轮工具场景要处理 `reasoning_content` 回传。
- Claude 4 系列强调 extended thinking、tool use、interleaved thinking、较成熟的 agent/coding 场景支持。
- Claude 4 的生态工具如 Claude Code、MCP connector、computer use、text editor tool 更偏 agentic workflow。
- DeepSeek 的优势常在性价比、开放兼容接口和中文体验；Claude 的优势常在复杂代码、长任务推理和工具生态。

**面试表达**

> 我会从三个维度比较：模型推理质量、工具调用接口、工程生态。DeepSeek V4 Pro 接入成本低，OpenAI-compatible 友好，但 thinking mode 要注意 reasoning_content 的多轮回传；Claude 4 在 agentic coding 和 extended thinking 生态上更完整。

### 07 怎么看待纯基座模型通吃一切和 LLM 工程架构派两种理念？

**回答要点**

- 基座模型越强，确实能减少 prompt、RAG、工具路由的一部分复杂度。
- 但工程系统仍需要权限、审计、状态、成本、延迟、数据治理和可观测性。
- 模型能力解决“能不能做”，工程架构解决“能不能稳定、可控、低成本地做”。

**面试表达**

> 我不认为两者矛盾。强模型降低了工程补丁的数量，但不会消灭工程架构。生产系统的关键不是让模型偶尔答对，而是让它在权限、成本、审计和失败恢复都受控的情况下持续工作。

---

### 08 LLM Harness 工程的核心设计理念是什么？

**回答要点**

LLM Harness 可以理解为围绕模型调用的一层工程外壳，负责：

- Prompt 模板和版本管理。
- 输入输出 schema。
- 模型路由、fallback、重试。
- 评测和回归。
- trace、token、cost、latency 统计。
- 安全策略和输出校验。

**面试表达**

> LLM Harness 的核心是把“调一次模型”变成“可配置、可评测、可观测、可回滚的工程组件”。它不追求替代模型，而是把模型调用纳入软件工程治理。

### 09 AI Coding 普及后，研发范式发生了哪些本质变化？

**回答要点**

- 开发者从“逐行写代码”转向“提出目标、约束、验收标准”。
- 代码生成成本下降，代码审查、架构判断、测试验证更重要。
- 文档、规则、测试、类型系统变成 AI 协作的接口。
- 小步迭代更快，但错误也可能规模化扩散。

**面试表达**

> AI Coding 不是让开发者消失，而是让开发者工作重心上移：从写实现细节，转向定义问题、设计边界、审查结果和建立自动化验证。

---

### 10 如果代码全部由 AI 生成，开发者核心需要聚焦哪些能力？

**回答要点**

- 问题定义能力：把需求拆成可执行任务。
- 架构判断能力：模块边界、状态、数据流、失败模式。
- 代码审查能力：安全、性能、可维护性、并发问题。
- 测试设计能力：单测、集成、回归、端到端验收。
- 工具链能力：让 AI 能运行、验证、回滚。

**面试表达**

> AI 生成代码后，开发者最核心的能力反而更像技术负责人：定义正确的问题，约束实现边界，建立验证体系，并对最终系统质量负责。

### 11 DeepSeek 和 Cursor 两类 AI 编程工具核心区别是什么？

**回答要点**

- DeepSeek 是模型/API 能力，提供推理和生成。
- Cursor 是产品化 IDE/Agent 工具，提供上下文管理、编辑器集成、规则、工具调用、MCP、代码修改体验。
- 模型解决智能能力，产品解决工作流和工程集成。

**面试表达**

> DeepSeek 更像发动机，Cursor 更像带方向盘、仪表盘和安全系统的车。做 AI 编程不能只看模型，还要看上下文、编辑、执行、测试和审查工作流。

### 12 DeepSeek 这类产品爆火的本质原因是什么？

**回答要点**

- 性价比高，降低开发者和企业试用门槛。
- OpenAI-compatible 接口降低迁移成本。
- 中文和本土生态友好。
- 推理模型/思考模式让复杂任务体验明显提升。
- 传播层面满足市场对低成本强模型的需求。

**面试表达**

> 本质原因不是单一模型指标，而是能力、成本和接入门槛同时改善。对开发者来说，能用较低成本接入较强推理能力，这会快速放大生态效应。

### 13 多人团队做 AI 项目，常见协作分工模式是怎样的？

**回答要点**

- 模型/Prompt 负责人：模型选型、prompt、评测。
- 后端/Runtime 负责人：API、状态机、任务执行、事件存储。
- 工具/集成负责人：Tool Registry、MCP、业务 API。
- 数据/RAG 负责人：知识库、索引、召回、数据治理。
- 前端/产品负责人：Chat UI、timeline、审批交互。
- QA/评测负责人：离线集、回归、线上监控。

**面试表达**

> AI 项目不是只有算法工程师。真正落地时，runtime、数据、工具、前端、评测和安全都要有人负责，否则系统很难从 demo 走向产品。

### 14 团队协作做 AI 项目容易出现哪些典型问题？

**回答要点**

- 只追模型效果，忽视数据、工具和评测。
- Prompt 改动不可追踪，导致效果回退。
- 工具缺少统一协议，返回格式混乱。
- 没有 trace，线上问题无法复盘。
- 评测集和真实业务脱节。
- 权限和安全后置，导致上线风险。

**面试表达**

> AI 项目最常见的问题是“感觉驱动开发”。解决方式是把 prompt、模型、工具、数据和评测都纳入版本管理和回归体系。

## 三面

### 01 多 Agent 任务执行选单行还是并行？常用实现方式是什么？

**回答要点**

- 默认串行，只有子任务独立、无强依赖、上下文不互相污染时才并行。
- 并行适合调研、测试、代码审查、多方案生成。
- 串行适合有严格依赖链的任务，例如先登录、再查询、再提交。
- 实现方式包括 asyncio task、消息队列、LangGraph parallel branches、任务队列 Celery/Temporal。

**面试表达**

> 不是多 Agent 就一定并行。并行的前提是任务之间低耦合、结果可合并、失败可局部处理。否则并行只会增加冲突和调试成本。

---

### 02 ReAct 架构下单个 Agent 反思轮数怎么设定，依据是什么？

**回答要点**

依据任务复杂度、工具调用成本、上下文长度和失败风险设定。

- 简单问答：0 到 2 轮。
- 一般工具任务：3 到 8 轮。
- 复杂排障/代码修改：10 到 20 轮，但必须有硬上限。
- 动态停止条件比固定轮数更重要：目标达成、信息不足、重复调用、成本超限、错误不可恢复。

**面试表达**

> ReAct 轮数不是越多越好。轮数上限是安全阀，真正决定是否继续的是状态：是否还有新信息、是否接近目标、是否重复失败、是否超过成本预算。

---

### 03 Agent 工具如何自研，怎么规划数量规模？

**回答要点**

- 先按业务任务反推工具，不按系统 API 全量暴露。
- 工具要小而清晰：单一职责、结构化参数、结构化返回。
- 每个工具注册 metadata：category、risk、timeout、approval、owner、version。
- 数量规划：单 Agent 可见工具尽量控制在十几个以内，大规模工具通过 router 动态召回。
- 高风险工具必须经过 policy 和 audit。

**面试表达**

> 工具自研的关键不是把接口包装成函数，而是设计工具契约。一个好工具应该让模型一眼知道何时使用、传什么参数、返回结果如何解释、失败怎么处理。

---

### 04 做 Agent 项目如何进行基座模型选型？

**回答要点**

从任务而不是榜单出发：

- 任务类型：问答、代码、工具调用、长上下文、多模态。
- 工具调用能力：function calling 稳定性、参数遵循。
- 成本和延迟：单位任务成本，而不是单 token 价格。
- 部署约束：云 API、本地、私有化、数据合规。
- 生态兼容：OpenAI-compatible、LangChain、SDK 支持。
- 评测结果：用业务样本跑 A/B。

**面试表达**

> 模型选型不能只看通用榜单。我会先定义业务评测集，再比较成功率、工具调用准确率、延迟、成本和稳定性，最后用可观测数据做决策。

---

### 05 大模型数据准确率初期偏低，有哪些通用优化思路？

**回答要点**

- 明确知识边界，不让模型猜。
- 引入 RAG，用权威数据源回答。
- 输出结构化，便于校验。
- 增加引用和证据链。
- 对关键字段做规则校验或二次验证。
- 建立错误样本集，持续回归。
- 对高频错误做 prompt、检索、工具或数据修复。

**面试表达**

> 准确率低不能只调 prompt。要先判断错误来自知识缺失、检索失败、推理错误、格式错误还是工具数据错误，然后分别优化。

---

### 06 Agent 架构如何设计来缓解大模型幻觉问题？

**回答要点**

- RAG：回答必须基于检索证据。
- Tool grounding：实时状态必须通过工具获取。
- Schema validation：结构化输出校验。
- Uncertainty policy：证据不足时允许拒答或请求更多信息。
- Citation：输出来源。
- Critic/Verifier：关键答案二次检查。
- Event timeline：记录模型看到的证据和动作。

**面试表达**

> 缓解幻觉的核心是减少模型自由编造空间。让模型基于证据、工具和 schema 工作，而不是凭记忆回答。

---

### 07 大模型领域 SFT 微调主要解决哪些通用问题？

**回答要点**

- 输出风格和格式稳定。
- 领域术语和任务模式适配。
- 指令遵循能力增强。
- 工具调用格式更稳定。
- 降低少样本 prompt 依赖。

不适合用 SFT 解决：

- 高频变化知识。
- 强事实一致性。
- 权限和安全边界。

**面试表达**

> SFT 更适合教模型“怎么答”和“按什么格式做”，不适合存储快速变化的事实。动态知识应该放 RAG 或工具里。

---

### 08 大模型 SFT 训练语料怎么构建，如何做质量过滤？

**回答要点**

- 构建：真实业务样本、专家答案、模型合成、人类修订。
- 覆盖：常见任务、边界情况、反例、拒答、工具调用。
- 过滤：去重、脱敏、格式校验、事实校验、长度过滤、低质回答过滤。
- 质量分层：gold/silver/bronze，不同权重训练。
- 保留独立测试集，不能被训练污染。

**面试表达**

> SFT 数据质量比数量重要。我的做法是先定义目标能力和数据 schema，再通过规则和人工抽检过滤，最后用独立评测集验证是否真的提升。

---

### 09 大模型与 Agent 测试重点关注哪些核心指标？

**回答要点**

大模型测试：

- 准确率、格式遵循、鲁棒性、安全性、成本、延迟。

Agent 测试：

- 任务成功率、工具调用准确率、状态机一致性、事件完整性、取消/审批、幂等性、回放能力。

**面试表达**

> LLM 测试更关注“回答对不对”，Agent 测试还要关注“过程可不可控”。因为 Agent 会调用工具、改变外部状态，所以测试必须覆盖执行链路。

### 10 成熟 Agent 系统有哪些通用优化方向？

**回答要点**

- Runtime：状态机、队列、取消、审批、恢复。
- Memory：摘要、召回、压缩、冲突处理。
- Tool：注册、路由、审计、权限、超时、重试。
- RAG：切分、召回、rerank、引用。
- Observability：trace、timeline、cost、latency。
- Evaluation：离线集、线上监控、回归。
- Security：PII、secret、sandbox、权限。

**面试表达**

> 成熟 Agent 的优化不是单点调 prompt，而是围绕 runtime、tool、memory、evaluation 和 observability 做系统工程。

### 11 LoRA 微调的核心原理是什么？

**回答要点**

LoRA 的核心是冻结原模型权重，只在部分线性层上添加低秩矩阵增量。原始权重更新可以近似为：

`W' = W + BA`

其中 `A` 和 `B` 是低秩矩阵，参数量远小于全量微调。

优点：

- 训练参数少，显存低。
- 多任务适配器可切换。
- 适合风格、格式、领域模式适配。

**面试表达**

> LoRA 本质是低秩增量学习。它不直接改动大模型主体，而是学习一个小的参数增量，因此成本低、可插拔、适合快速适配。

---

### 12 LoRA 秩超参如何选型，不同秩会带来什么影响？

**回答要点**

- rank 越小，参数少、训练快、过拟合风险低，但表达能力有限。
- rank 越大，表达能力更强，但显存、训练时间和过拟合风险增加。
- 常见从 8、16、32、64 做网格实验。
- 简单格式/风格任务用低 rank，复杂领域能力迁移用较高 rank。
- 选型要看验证集，不凭经验拍脑袋。

**面试表达**

> rank 是容量和成本的权衡。我的做法是从较小 rank 开始，通过验证集观察收益是否饱和，如果提升不明显就不继续加。

---

### 13 多业务交叉场景，Agent 怎么做意图分发与任务调度？

**回答要点**

- 先做意图识别：业务域、任务类型、风险等级。
- 再路由到对应 workflow/agent/tool set。
- 对复杂任务拆子任务，形成 plan。
- 执行层维护 run 状态，支持并发限制、取消、审批。
- 结果聚合后统一输出。

**面试表达**

> 多业务场景不要让一个 Agent 直接面对所有工具。应先做 intent routing，把任务分到对应业务域，再加载对应工具和规则，最后由 runtime 调度执行。

### 14 近期接触印象最深的大模型新技术、新特性有哪些？

**回答要点**

可以回答：

- Extended thinking / thinking mode：模型在复杂任务前进行更长推理。
- Interleaved thinking with tools：工具调用之间继续推理。
- MCP：工具和上下文标准化接入。
- Agentic coding：模型能读代码、改文件、运行测试、回看结果。
- Long context 和上下文压缩。
- Structured output 和 function calling 更稳定。

**面试表达**

> 我印象最深的是 thinking mode 与工具调用结合。它把模型从一次性回答推进到“边思考、边调用工具、边修正”的执行模式，但工程上也带来新要求，比如 reasoning 内容回传、上下文预算和工具审计。

### 15 日常工作中如何搭配使用各类 AI 编程辅助工具？

**回答要点**

- 快速问答和方案比较：通用聊天模型。
- 代码库内修改：Cursor/Claude Code/Codex 类 Agent。
- 文档和 API 查询：RAG、MCP docs、官方文档。
- 批量重构：CLI agent + 测试回归。
- 代码审查：让 AI 做第一轮风险扫描，人做最终判断。
- 规则沉淀：把项目规范写成 Rules/AGENTS.md。

**面试表达**

> 我会按任务选择工具：设计阶段用模型做方案推演，编码阶段用 IDE/CLI agent 做小步修改，验证阶段必须跑测试和 review。AI 是加速器，但最终质量要靠工程闭环保证。
